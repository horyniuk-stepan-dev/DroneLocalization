## 1. Перша черга — дні, без нових залежностей

### 1.1. Виправити токени DINOv3 у `forward_features()` + сітку патчів
**Файли:** `src/models/wrappers/dinov3_wrapper.py`, `src/models/wrappers/feature_extractor.py:81,115`, `src/database/database_builder.py:274`.

- У DINOv3 layout `last_hidden_state` = [CLS, register-токени, патчі]. Зрізати `hidden[:, 1 + n_reg :, :]`, де `n_reg = self._model.config.num_register_tokens` (читати з конфігу моделі в рантаймі, не хардкодити 4).
- Сітку патчів рахувати як `dino_size // self._model.config.patch_size` (16 для DINOv3), а не `// 14`.

Зараз CESP вимкнено (`models.cesp.enabled=false`), тож продакшн-CLS-гілка не зачеплена — але це **обов'язковий пререквізит** для VLAD (п. 2.1), і латентний баг для будь-кого, хто ввімкне CESP на DINOv3.

### 1.2. ~~Робастна loss у pose-graph оптимізації~~ — СКАСОВАНО
**Виявлено при імплементації:** `config/graph.py` документує, що robust loss
(`soft_l1`) уже пробували і свідомо прибрали — spatial-ребра (loop closures) мають
природно великі резидуали, і глобальна loss пригнічувала саме їх («якорі тримали
свої кадри, а решта пливла»). Замість неї в системі вже є клас-свідомий GNC
(Geman-McClure з σ у межах spatial-класу, `gnc_spatial`) та two-stage MAD-prune.
Глобальна scipy-`loss` не розрізняє класи резидуалів і була б регресом. Рекомендацію
блоку 4 джерел («робастне ядро на фактори») система вже покриває точніше за статті.

### 1.3. Інтерполювати log(s), а не s
**Файли:** `src/calibration/multi_anchor_calibration.py:188-198`, `src/workers/calibration_propagation_worker.py:927`.

Єдиний геодезично некоректний залишок у наявній 5-DoF інтерполяції: лінійна інтерполяція `sx, sy` замість `log(sx), log(sy)`. Зміна — по 4 рядки на місце (log при декомпозиції, exp при композиції). Це весь практичний зміст блоку 3 документа для цього коду; тягнути Sophus/gtsam Expmap заради 2D-афінних перетворень не потрібно.

## 2. Друга черга — тижні

### 2.1. VLAD-агрегація патч-токенів (AnyLoc) за фіче-флагом
**Пререквізит:** 1.1. **Файли:** новий `src/models/wrappers/vlad_aggregator.py`, `src/database/database_builder.py`, `config/models.py`.

- Офлайн: патч-токени з проміжного шару всіх референс-тайлів → k-means (32 кластери) → VLAD-залишки → intra-norm → L2 → PCA-whitening до 512 → новий FAISS-індекс. Проміжний шар — гіперпараметр (документ: ~31 для ViT-L; підбирати валідацією, бо AnyLoc міряв DINOv2, не DINOv3-SAT).
- Онлайн: та сама агрегація на query. Сурогат SALAD-dustbin задарма: перед агрегацією викидати патчі під YOLO-маскою динамічних об'єктів (маска вже є в пайплайні) та патчі з низькою нормою.
- Розмірність зміниться (1024 → 512) — база потребує перебудови; тримати CLS-гілку за флагом для A/B на Flightmare-футажі. Увага: пайплайн уже має альтернативні гілки patchify/CESP — VLAD додавати як третю опцію того ж інтерфейсу, не як новий шар.

### 2.2. SIFT+LightGlue аварійний фолбек
**Файли:** `src/localization/matcher.py`, `src/models/model_manager.py`, `src/localization/localizer.py`.

Тригер: `best_inliers < min_matches` після основного ALIKED-проходу (точка вже існує — `localizer.py:349`, зараз там одразу фолбек на optical flow). Одноразовий перезапуск кадру: OpenCV SIFT-екстракція + окремий інстанс LightGlue `features="sift"` (підтримується нативно). Це страхує in-plane rotation і екстремальну похилість — слабкість ALIKED, підтверджену джерелами [5][6]. Але: **база теж має зберігати SIFT-дескриптори** для референс-тайлів (окреме поле в БД або обчислення на льоту для top-1 кандидата — друге простіше і достатнє для аварійного шляху). Міряти частоту спрацювання; якщо <1% кадрів — вигода сумнівна проти складності.

### 2.3. Yaw-вхід — лише як опційний параметр
Не будувати телеметричний тракт заради цього. Додати опційний `yaw_hint` у `localize_frame()`: якщо заданий (симулятор його має — CalibrationLogger v2.3), кадр повертається до north-up перед екстракцією, `RotationSelector` пропускає скан. Реальні польоти без телеметрії продовжують жити на A3-prior.

## 3. Третя черга — найбільший резерв точності

### 3.1. Ковзний віконний smoother для live-траєкторії
Зараз live-трекінг = `TrajectoryFilter` (KF) + `OutlierDetector` (Z-score, бінарне прийняти/відкинути). Граф поз використовується лише офлайн у калібруванні. Архітектурний патерн з [11][12] застосовний так:

- **Front-end:** наявний KF, покадровий вивід без змін.
- **Back-end:** асинхронний потік, ковзне вікно 50–100 поз; вузли — 2D-пози ключових кадрів; ребра — міжкадрова відносна одометрія (потік optical flow вже є); абсолютні візуальні фікси — м'які prior-фактори з робастною вагою (Huber з 1.2). Хибний фікс отримує пригнічену вагу замість бінарного відкидання — страхує саме те, що Z-score пропускає.
- **Реалізація:** перевикористати власний `PoseGraphOptimizer` (він уже вміє м'які якорі `add_anchor()` і зважені ребра) в конфігурації ковзного вікна, а не тягнути GTSAM: відома проблема CustomFactor для 5-DoF задокументована в `scripts/allFiles/all_merged_new.py:3932`. GTSAM/iSAM2 має сенс лише якщо вікно не влазить у бюджет кадру — заміряти спершу scipy-варіант.
- Корекція стану KF — періодична, через зсув; запізнілі фікси (SIFT-фолбек з 2.2) додаються до історичної пози вікна природно.

### 3.2. Методологія оцінювання (наскрізно)
- Метрика PDM@K у тестовий контур (зв'язує якість retrieval з фінальною похибкою — дешева, рахується з уже логованих величин).
- Деградована референс-гілка: та сама місцевість із супутникових тайлів замість Cesium-рендера. Розрив 74.1% vs 18.5% з [2] — найбільший відомий фактор якості; без цієї гілки всі A/B-цифри на Flightmare-футажі систематично оптимістичні.

## 4. Не впроваджувати

- **Top-5 ре-ранкінг** — уже є (top_k=8). Єдине можливе покращення: батчувати матчинг кандидатів замість послідовного циклу з early-stop; міряти профайлером, чи цикл взагалі у топі витрат.
- **SALAD** — згоден з документом: ваги прив'язані до DINOv2 і наземного домену; без аеродатасета для донавчання сенсу немає.
- **GTSAM Pose3/iSAM2 «за специфікацією»** — конфліктує з 5-DoF анізотропною моделлю системи (не SE(3)-задача). Робастність дає п. 1.2, вікно — п. 3.1.
- **Sophus / gtsam Expmap для інтерполяції** — для 2D-афінних перетворень decompose-interpolate-compose з log-scale (п. 1.3) математично еквівалентний і вже на 90% написаний.
- **IndexHNSWFlat** — точний IndexFlatIP на поточних розмірах бази швидкий і без recall-втрат.

## 5. Виправлення до самого документа-аналізу

1. `OutlierDetector` — Z-score за швидкістю, не MAD (MAD — у RANSAC і prune графа).
2. `GtsamPoseGraphOptimizer` — не «заглушка в системі», а експеримент у `scripts/allFiles/` з задокументованою причиною відмови.
3. «Наївна інтерполяція» — калібрувальна інтерполяція вже 5-DoF компонентна з unwrap кутів; наївною вона не є.
4. Ре-ранкінг і поріг inliers (обидві «дії» з розбору AnyVisLoc) — уже в коді.
5. FAISS — `IndexFlatIP`, не HNSW.
