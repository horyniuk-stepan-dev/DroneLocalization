## 0. Профіль теми, за яким шукалось

Задача проєкту в літературі називається **Absolute Visual Localization (AVL)** для БпЛА в GNSS-denied умовах. Але «AVL» — це широкий клас; твоя система займає в ньому специфічну точку. Саме за цими вісями шукались сусіди:

| Вісь | Твоя система | Як зазвичай у літературі |
|---|---|---|
| Джерело реферної карти | **власне відео попереднього обльоту** → БД дескрипторів | супутникові тайли / ортофото / DSM / 3D-модель / OSM |
| Геоприв'язка цієї карти | **кілька ручних GPS-якорів + 5-DoF pose-graph пропагація** | карта вже геоприв'язана (публічні геодані) або фотограмметрія з GCP |
| Вхід | відеопотік: keyframes + optical flow між ними | окремі кадри або відео з VIO/IMU |
| Retrieval | заморожений VFM (DINOv3-sat), CLS/VLAD → LanceDB | навчені агрегатори (NetVLAD/SALAD/BoQ), крос-в'ю сіамські мережі |
| Matching | sparse: ALIKED/RDD + LightGlue | sparse / semi-dense (LoFTR) / dense (RoMa) |
| Геометрія | гомографія → 5-DoF анізотропна афінна → GPS | 3-DoF/4-DoF similarity, гомографія, або 6-DoF PnP по DSM |
| Ф'южн | KF + Huber fixed-lag smoother, **одна гіпотеза** | particle filter / фактор-граф / EKF з IMU |
| Сенсори | **тільки камера** (телеметрії немає) | камера + IMU (+ барометр, магнітометр, лазерний висотомір) |
| Розгортання | десктоп GUI + headless, дискретна NVIDIA | бортовий edge (Xavier NX / Orin), онлайн |

Дві вісі роблять тему нішевою: **джерело карти** (власний обліт, а не публічні геодані) і **спосіб її геоприв'язки** (пропагація розріджених якорів). Все інше — мейнстрім AVL.

## 1. Рівень 0 — архітектурні близнюки

Роботи, де збігається не лише задача, а й декомпозиція системи. Це і є «найближчі дослідження».

### 1.1. OrthoTrack — найближчий аналог загалом
**ECCV 2026, [arXiv:2606.25245](https://arxiv.org/abs/2606.25245)** · training-free

Матчить **keyframes** проти публічного ортофото, піднімає відповідності в метричний 3D через DSM, і **пропагує map-anchored відповідності на проміжні кадри оптичним потоком** — абсолютні метричні пози на кожному кадрі, без GPS і без post-hoc вирівнювання. Реалтайм на одному GPU. Разом із роботою випущено датасет MovingDrone (фотореалістичні UAV-послідовності + щільний 6-DoF GT + ко-реєстровані мультитемпоральні геодані).

- **Збіг:** схема «розріджені абсолютні фікси на keyframes + OF-пропагація між ними» — це буквально твій `Localizer` + OF-шов. Плюс training-free підхід (готові моделі, без донавчання) — теж твій.
- **Різниця:** карта — публічне ортофото + DSM (без обльоту), поза — 6-DoF через DSM замість планарної гомографії, матчер — dense.
- **Що це означає для тебе:** це найсильніший baseline для порівняння і водночас найгостріша конкуренція твоєму «спочатку зроби обліт». Датасет MovingDrone — готовий полігон з 6-DoF GT.

### 1.2. AerialVL — та сама модульна декомпозиція
**IEEE RA-L 9(10):8210–8217, 2024** · [код](https://github.com/hmf21/AerialVL)

Датасет + baseline + **уніфікований модульний фреймворк** аеро-локалізації з реферною картою; оцінює три модулі окремо: visual place recognition, visual alignment localization, visual odometry. ~70 км траєкторій, 11 послідовностей, 18 361 аерокадр + 14 096 кропів карти.

- **Збіг:** розбиття «VPR → вирівнювання → VO» = твоє «retrieval → гомографія → OF». Найближчий готовий каркас для покомпонентного A/B.
- **Різниця:** реферна база — супутникова.

### 1.3. SatLoc / SatLoc-Fusion — той самий трьохшаровий ф'южн
**Remote Sensing 17(17):3048, 2025** · [код](https://github.com/ameth64/SatLoc)

Три шари: абсолютна геолокалізація по супутнику на **DINOv2**, високочастотний відносний трекінг (VO на XFeat), оцінка швидкості оптичним потоком; **адаптивне зважування шарів за онлайн-confidence**. На 6-TFLOPS edge — >2 Гц, абсолютна похибка <15 м, покриття траєкторії >90%.

- **Збіг:** DINOv2-retrieval + VO + OF і зважений ф'южн — покомпонентно те саме, що в тебе.
- **Що взяти:** їхня схема адаптивних ваг за confidence — пряма альтернатива твоєму бінарному outlier-гейту; і метрика «effective trajectory coverage» варта того, щоб з'явитись у твоєму тестовому контурі.

### 1.4. FoundLoc — попередник твого вибору бекбона
**CMU Robotics Institute, [arXiv:2310.16299](https://arxiv.org/abs/2310.16299)**

VIO + VPR на foundation-моделі (AnyLoc), надирна камера + IMU + наявні супутникові знімки, без припущень про початкову позу. Середня похибка в межах 20 м; VPR-гілка ~1.93 Гц на Xavier NX.

- **Збіг:** ідея «заморожений VFM як глобальний дескриптор для аеролокалізації» — це рівно твоя retrieval-гілка (і рівно те, що в `SOTA_RESEARCH.md` названо відставанням: FoundLoc — 2023-й).
- **Різниця:** є IMU/VIO, карта супутникова.

### 1.5. Hierarchical Image Matching for UAV AVL
**[arXiv:2506.09748](https://arxiv.org/abs/2506.09748)** (анотація звірена)

Retrieval → грубий семантично-структурний матчинг на фічах vision foundation model (регіональні відповідності) → легкий fine-матчинг (піксельні відповідності). Явно **без відносної локалізації** — тільки абсолютні фікси. Свій датасет CS-UAV.

- **Збіг:** триступенева схема retrieval→coarse→fine і опора на VFM.
- **Різниця:** VFM використано і для матчингу, не лише для retrieval; у тебе між VFM-retrieval і ALIKED-матчингом семантичного шару немає — це найдешевша ідея звідси.

## 3. Рівень 2 — суміжні постановки (корисні як донори ідей)

- **Cross-view geo-localization** — окрема гілка: University-1652 / SUES-200 / Game4Loc / UAV-VisLoc ([arXiv:2405.11936](https://arxiv.org/abs/2405.11936)). Свіже: [Unifying UAV Cross-View Geo-Localization via 3D Geometric Perception](https://arxiv.org/abs/2604.01747) (анотація звірена) — реконструює локальну 3D-сцену з послідовності кадрів через VGGT, рендерить віртуальний BEV і ним ортовирівнює вид перед retrieval-ом; цей BEV одночасно дає просторовий prior для 3-DoF регресії пози. Ідея «ортовирівняти запит перед retrieval» — прямий конкурент твоєму скану 4 ротацій.
- **[Beyond Paired Data / CAEVL](https://arxiv.org/abs/2512.02737)** (WACV 2026, анотація звірена) — навчання **тільки на реферних знімках**, без жодного UAV-кадру: аугментації імітують доменний зсув «карта → реальний вид». Це найдешевший спосіб реалізувати п. 6 твого roadmap (навчена агрегаційна голова) — позитиви/негативи генеруються з самої БД, збирати парні дані не треба.
- **SASGeo** ([arXiv:2607.07737](https://arxiv.org/abs/2607.07737), липень 2026) — локалізація по **семантичній карті** (дороги, будівлі, водойми, перехрестя, межі полів) зі stability-aware вагами й явним відкиданням неоднозначних фіксів. Глобальний семантичний дескриптор дав R@1 58.6%, просторово-семантичні варіанти — 94.5–95.5% (синтетичний proof-of-concept, не реальний політ). Цінне саме integrity-aware відкидання: твій `out_of_coverage` — груба версія того самого.
- **TanDepth** ([arXiv:2409.05142](https://arxiv.org/abs/2409.05142), IEEE 2025) — відновлення метричного масштабу для будь-якої відносної моделі глибини через глобальні DEM (TanDEM-X). У тебе вже стоїть Depth-Anything-V2 з відносною глибиною — це готовий рецепт, як зробити її метричною без нового сенсора.
- **Семантичне маскування нестабільних областей** — «Semantic-aware visual localization for robust aerial perception» (Aerospace Sci&Tech 2025) і [Semantic Masking and Visual Feature Matching](https://arxiv.org/abs/2411.01804): матчинг спрямовується на темпорально стабільні класи, пригнічуються рослинність і динаміка. Твоя YOLO-маска ловить тільки динаміку — рослинність (найбільше джерело сезонного дрейфу) лишається.
- **Карти-абстракції замість знімків** — [InterKey](https://arxiv.org/abs/2509.13857) (перехрестя з OpenStreetMap як крос-модальні ключові точки), road-network alignment. Клас карти, який не старіє і важить мегабайти.
- **LoD-Loc v2** ([arXiv:2507.00659](https://arxiv.org/abs/2507.00659)) — локалізація по грубих LoD-моделях міста через вирівнювання силуетів. Актуально, якщо колись буде місто.

## 4. Де в літературі дірка — і що з цього твій внесок

**Твоя стадія побудови БД не має прямого аналога в AVL-літературі.** Роботи діляться на два табори:

1. **Карта — публічні геодані** (супутник, ортофото, DSM, 3D-модель, OSM): AnyVisLoc, OrthoTrack, OrthoLoC, LSVL, FoundLoc, SatLoc, SWA-PF, GROMS. Геоприв'язка дана задарма, питання лише в крос-доменному розриві.
2. **Карта — фотограмметричний продукт**, побудований офлайн з GCP (Metashape-клас) або SfM/SLAM з bundle adjustment під GCP-констрейнтами.

Твоя схема — третій варіант: **реферна база будується з сирого відео обльоту, а геоприв'язка вводиться кількома ручними GPS-якорями й розповзається по графу поз 5-DoF пропагацією**. Це «легка фотограмметрична геоприв'язка без фотограмметричного пайплайна». Найближчі родичі — не AVL-роботи, а (а) GCP-constrained bundle adjustment у фотограмметрії, (б) GNSS-anchored pose graph у SLAM, (в) Visual Teach & Repeat (карта з навчального проходу — але VT&R дає **відносну** локалізацію вздовж маршруту, без геокоординат).

Наслідки, і приємні, і ні:

- **Це захищабельна новизна** для статті/дисертації: постановка «мінімум ручних якорів → геоприв'язана аеро-БД → drift-free локалізація» в оглядах AVL не виділена в окремий клас.
- **Її ніхто не міряє.** Жоден із бенчмарків вище не оцінює якість геоприв'язки карти, побудованої користувачем — усі приймають карту як даність. Метрику для цієї стадії доведеться визначати самому (напр. leave-one-anchor-out: похибка реконструкції відкинутого якоря як функція щільності якорів).
- **Другий нетиповий елемент — 5-DoF анізотропна афінна модель.** Література працює або в 3-DoF/4-DoF similarity (SWA-PF), або в гомографії, або в повних 6-DoF по DSM. Проміжний 5-DoF варіант доведеться обґрунтовувати явно: він виправданий саме тим, що реферна карта — не ортофото, а перспективні кадри іншого обльоту.
- **Слабке місце в порівнянні:** твоя схема вимагає попереднього обльоту, а OrthoTrack і GROMS показують, що на публічних геоданих уже виходить реалтайм і метричний масштаб. Питання «навіщо обліт, якщо є ортофото» треба закривати даними (точність, зони без свіжого покриття, автономність від зовнішніх сервісів), а не аргументом зручності.

## 6. Чим міряти, щоб порівняння було чесним

- **Датасети:** AnyVisLoc (аеро + супутникові 2.5D карти), AerialVL (реферна карта + послідовності), MovingDrone (щільний 6-DoF GT, з OrthoTrack), UAV-VisLoc, SatLoc, MAFS (змінна висота, зі SWA-PF), UAVD4L/OrthoLoC (oblique, 6-DoF).
- **Метрики, які вживає ця література:** A@Xm (частка кадрів у межах X метрів — де-факто стандарт AVL), R@1/AP для retrieval, RMSE траєкторії, **effective trajectory coverage** (частка кадрів із валідним фіксом — у тебе ця величина логується, але як метрика не подана).
- **Обов'язковий контроль:** деградована реферна гілка (та сама місцевість із супутникових тайлів). Розрив між «аерокарта» і «супутникова карта» в AnyVisLoc — найбільший відомий фактор точності; без цієї гілки всі твої цифри систематично оптимістичні. Це вже записано в `RESEARCH_INTEGRATION_PLAN.md` п. 3.2 і досі не зроблено.
